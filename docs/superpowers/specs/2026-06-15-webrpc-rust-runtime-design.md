# WebRPC Rust Runtime Design

## Scope

Add a Rust WebRPC runtime path to `lyre-web` that serves the existing `proto/lyre.ridl` service over Axum at the generated TypeScript client's expected URLs:

- `POST /rpc/Lyre/GetRoom`
- `POST /rpc/Lyre/JoinRoom`
- `POST /rpc/Lyre/LeaveRoom`
- `POST /rpc/Lyre/GetNoiseProviders`
- `POST /rpc/Lyre/GetIceServers`
- `POST /rpc/Lyre/GetMediaTopology`
- `POST /rpc/Lyre/GetMediaRelay`
- `POST /rpc/Lyre/StartMediaRelay`
- `POST /rpc/Lyre/StopMediaRelay`
- `POST /rpc/Lyre/RegisterMediaTrack`
- `POST /rpc/Lyre/AnswerServerMediaOffer`
- `POST /rpc/Lyre/AddServerMediaIceCandidate`
- `POST /rpc/Lyre/GetServerMediaIceCandidates`
- `POST /rpc/Lyre/CloseServerMediaSession`

This increment does not replace the existing REST endpoints, does not switch the frontend helpers to WebRPC by default, does not add streaming WebRPC, and does not create a full general-purpose Rust RIDL code generator. The official WebRPC toolchain already generates the TypeScript client used by this repo; this increment adds a repo-local Rust runtime module that is kept aligned with the checked-in RIDL schema and generated TypeScript client.

## Goals

- The committed generated TypeScript `Lyre` client can call the Rust API server through `/rpc/Lyre/*`.
- WebRPC methods reuse the same `AppState` mutation and validation paths as REST where possible, so join/leave persistence, metrics counters, media relay state, and server-media session behavior stay consistent.
- REST endpoints remain stable and continue to pass existing tests.
- WebRPC JSON uses the field names from the RIDL/generated TypeScript contract, such as `roomID`, `accessToken`, `audioTrackID`, `sdpMid`, `sdpMLineIndex`, and `usernameFragment`.
- WebRPC responses wrap values exactly like the RIDL service signatures, such as `{ "room": ... }`, `{ "mediaRelay": ... }`, `{ "answer": ... }`, `{ "accepted": ... }`, and `{ "closed": ... }`.

## Compatibility

The TypeScript generated client always sends `POST` requests to `/rpc/Lyre/<Method>` with:

- `Content-Type: application/json`
- a `Webrpc` header value generated by `frontend/src/lib/lyre.gen.ts`
- a JSON request body, or `{}` for no-input methods

The Rust runtime should not require or validate the exact `Webrpc` header value in this increment. It should accept generated-client requests and ordinary JSON POST tests. `GET` or other methods are not added for RPC routes.

## Authorization

Methods that mutate or read user-scoped protected state must preserve the REST authorization model by accepting the same `Authorization: Bearer <token>` header:

- `LeaveRoom`
- `StartMediaRelay`
- `StopMediaRelay`
- `RegisterMediaTrack`
- `AnswerServerMediaOffer`
- `AddServerMediaIceCandidate`
- `GetServerMediaIceCandidates`
- `CloseServerMediaSession`

Public discovery/read methods remain unauthenticated:

- `GetRoom`
- `JoinRoom`
- `GetNoiseProviders`
- `GetIceServers`
- `GetMediaTopology`
- `GetMediaRelay`

`JoinRoom` returns `accessToken` only in its successful WebRPC response. No other WebRPC response should include room access tokens.

## DTO Mapping

Add WebRPC-facing DTOs in a focused `lyre-web` module, not by changing the serde names of existing core or REST DTOs.

Required enum string values must match generated TypeScript enum values:

- `NoiseProvider`: `OFF`, `RNNOISE`, `DEEPFILTERNET`
- `MediaTopologyMode`: `P2P_MESH`, `MEDIA_RELAY`
- `MediaRelayStatus`: `INACTIVE`, `ACTIVE`
- `MediaRelayMode`: `P2P_MESH`, `MEDIA_RELAY`
- `MediaTrackKind`: `AUDIO`, `VIDEO`
- `ServerMediaSessionState`: `NEW`, `NEGOTIATING`, `CONNECTED`, `CLOSED`

Required field-name conversions:

- Rust/core REST `room_id` <-> WebRPC `roomID`
- `user_id` <-> `userID`
- `joined_at` <-> `joinedAt`
- `voice_activity_threshold` <-> `voiceActivityThreshold`
- `track_id` <-> `trackID`
- `audio_track_id` <-> `audioTrackID`
- `access_token` <-> `accessToken`
- `media_relay` <-> `mediaRelay`
- `sdp_mid` <-> `sdpMid`
- `sdp_mline_index` <-> `sdpMLineIndex`
- `username_fragment` <-> `usernameFragment`

## Error Behavior

Return WebRPC-compatible JSON errors for failed RPC calls:

```json
{
  "name": "WebrpcEndpoint",
  "code": 0,
  "message": "...",
  "status": 400
}
```

The HTTP status should match the existing `ApiError` status mapping where practical:

- Invalid room IDs and malformed JSON: `400`
- Unauthorized access token: `401`
- Media relay conflicts: `409`
- Persistence and unexpected server-media internal failures: `500`

Persistence errors must still log the full underlying error chain server-side and return the generic public message `room state persistence failed`.

Public WebRPC error envelopes must be sanitized:

- Unauthorized access returns `room access token is invalid`.
- Malformed JSON returns `bad request`.
- Invalid room IDs return the validation message and never include raw access tokens or media payloads.
- Media relay conflicts may return the existing conflict message because it contains room/user/track identifiers already supplied by the caller and no secrets.
- TURN credential failures return the existing non-secret TURN validation message.
- Server-media/WebRTC failures return a generic public message such as `server media negotiation failed`; full lower-level SDP, ICE, WebRTC, or transport cause chains must be logged server-side with context and not emitted in the WebRPC envelope.
- Persistence failures return `room state persistence failed` and log the full underlying error chain server-side.

No public WebRPC error envelope may include access tokens, SDP bodies, ICE candidate strings, RTP payloads, raw audio samples, persistence file paths, or lower-level OS/network error chains.

## Module Layout

Keep the already-large REST API files from growing substantially:

- Add a new `crates/lyre-web/src/webrpc.rs` module for WebRPC DTOs, route wiring, handlers, conversions, and tests if concise enough.
- If the module approaches the 400 LOC threshold, split tests into `crates/lyre-web/src/webrpc_tests.rs` and keep route/runtime code separate.
- Add only a `.merge(crate::webrpc::router())` call in the existing `api::router`.
- Reuse existing `AppState` methods and authorization helpers; do not duplicate persistence or media relay state machines.

## Tests

Add Rust tests covering:

- `JoinRoom` through `/rpc/Lyre/JoinRoom` returns camelCase `accessToken`, `room.roomID`, `user.joinedAt`, and uppercase enum provider values.
- `GetRoom` returns the joined user through WebRPC wrapper shape `{ room: ... }`.
- `LeaveRoom` requires a bearer token and succeeds with the token returned by `JoinRoom`.
- `GetNoiseProviders`, `GetIceServers`, `GetMediaTopology`, and `GetMediaRelay` return RIDL wrapper shapes and enum values.
- Media relay WebRPC methods can start a relay, register an audio track, and stop the relay using bearer auth.
- Server-media WebRPC methods return successful RIDL wrapper shapes where a valid test offer/candidate path is available:
  - `AnswerServerMediaOffer` returns `{ "answer": { "roomID", "userID", "audioTrackID", "sdp", "state" } }`.
  - `AddServerMediaIceCandidate` returns `{ "accepted": { ... } }`.
  - `GetServerMediaIceCandidates` returns `{ "candidates": [ ... ] }`.
  - `CloseServerMediaSession` returns `{ "closed": { "mediaRelay": ..., "session": ... } }`.
- Server-media protected RPC methods reject missing bearer tokens with sanitized WebRPC error envelopes.
- Invalid room IDs and malformed JSON return WebRPC error envelopes, not the REST `{ "error": ... }` shape.
- Server-media failures do not echo SDP or ICE candidate strings in public WebRPC errors.
- Unknown `/rpc/Lyre/<Method>` routes return 404.

Existing REST tests must continue passing.
